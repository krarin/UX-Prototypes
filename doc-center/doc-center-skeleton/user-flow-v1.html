Edge-case loops

E1
After Neue Version anfordern

Document moves to abgelehnt. Customer uploads a fresh version → the auto-step "Digital QA runs checks" runs again → entire decision tree re-evaluates from the top.

E2
After Ergänzendes Dokument anfordern

Sub-slot waits for customer (hourglass in sidebar). Customer uploads → QA checks the sub-slot → if sub-slot approved, the triggering Red is automatically resolved. When the last Red is resolved, the parent doc enters the approval path.

E3
All Reds resolved via Manuell freigeben

No requests, no new version needed — all Reds are overridden in one session. Advisor clicks Genehmigen ✓ directly. Each Begründung stays attached to its row as an audit trail.

E4
Mixed state — Reds + Ambers together

Document stays blocked as long as any Red is unresolved. Amber actions (notes / requests) can run in parallel but do NOT change the Approve state. Resolving Reds is the only path to enabling Genehmigen ✓.

E5
After approval — revert

Once a document is approved, it's locked. If the advisor uses Zurücksetzen on an individual Manuell freigegeben row and that reintroduces a Red, the Approve state automatically resets to blocked. Sidebar checkmark reverts. The advisor then handles the Red and re-approves.

info Notes for the team

  • This flow is documentation only — V1–V4 prototypes implement parts of it but not all. Notable drift: V3/V4's Genehmigen ✓ button is currently not disabled while Reds are unresolved. This is flagged as future work.
  • Manuell freigeben is intentionally not available on Amber rows in this refined model — Amber doesn't block, so there's nothing to override. (V3/V4 still expose it; that's an inconsistency to clean up.)
  • The X button on the document toolbar is the only way to trigger Neue Version anfordern. It's always available, regardless of check state.
  • Customer notes ride a separate lifecycle: in V3 they ship immediately, in V4 they're collected as Entwurf and bundled with one Senden click.
